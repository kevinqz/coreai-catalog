# Contributing to Core AI Catalog

Thank you for helping build a more complete and trustworthy catalog of Apple Core AI models.

## Source of truth

YAML/JSONL data files are authoritative. Markdown docs, JSON exports, and the
entry templates are derived views generated by scripts. **Always edit the
source files, never edit generated files directly.**

| Edit directly | Generated (don't edit) |
|---|---|
| `catalog.yaml` | `docs/model-registry.md`, `docs/artifact-provenance.md` |
| `artifacts.yaml` | `docs/apple-terminology-map.md`, `docs/index.md` |
| `sources.yaml` | `docs/compare/` (per-capability comparison pages) |
| `upstreams.yaml` | `docs/tasks/` + `dist/tasks/` (task pages) |
| `benchmarks.jsonl` | `dist/*.json`, `dist/models.jsonl` (all JSON exports) |
| `terms.yaml` | `templates/` (rendered from `schema/*.json` by `scripts/generate_templates.py`) |
| `schema/*.json` | |

> Benchmarks live **only** in `benchmarks.jsonl` (append-only, one JSON object
> per line). The legacy YAML benchmark store is retired — never add benchmark
> data anywhere else.

## Two contribution lanes — never mix them

| Lane | Files touched | Review path |
|---|---|---|
| **Model lane** | `catalog.yaml`, `artifacts.yaml`, `sources.yaml`, `upstreams.yaml` + regenerated `docs/`/`dist/` | Human maintainer review |
| **Benchmark lane** | `benchmarks.jsonl` **only**, exactly **one added line** per PR | Signed relay auto-merge, or curator review (see below) |

A model PR must **never** touch `benchmarks.jsonl`. Any PR that touches
`benchmarks.jsonl` triggers `benchmark-validate.yml`, which enforces the
one-line rule and the lane rules below — mixing a model addition with a
benchmark row makes the PR unmergeable. Split it into two PRs.

## Add a model

Start from the generated templates in `templates/` (`model-entry.yaml`,
`artifact-entry.yaml`). They are rendered directly from the JSON Schemas, so
the field comments and enum lists are always exact.

### 1. Add model metadata to `catalog.yaml`

Append an entry under `models:`:

<!-- doc-test: model -->
```yaml
- id: my-model-1b
  name: MyModel-1B
  family: MyModel
  source_group: zoo          # zoo | official | external (see schema for the full enum)
  source_path: https://github.com/.../model-card.md
  artifact_ref: my-model-1b   # must match an id in artifacts.yaml
  capabilities:
    - chat
    - text-generation
  modalities:
    input:
      - text
    output:
      - text
  artifact:
    format: aimodel
    availability: available
  size:
    parameters: 1B
    precision: int8
    quantization: int8lin
    artifact_size: 850MB
  runtime:
    runtime_name: apple-core-ai
    runner: CoreAIRunner
    stock_runtime: false
    custom_kernel: false
    patch_required: true
    tokenizer_required: true
    processor_required: false
    aot_required: false
  device_support:
    iphone: true
    ipad: unknown
    mac: true
    mac_only: false
  license:
    name: Apache-2.0
    commercial_use: likely      # likely | check_license
  status: confirmed              # confirmed | needs_review | deprecated | unknown
  maturity: experimental         # stable | active | experimental | research | unknown
  confidence: medium             # high | medium | low | needs_review
  sources:
    - coreai-model-zoo-readme
    - coreai-model-zoo-zoo
  last_verified: '2026-06-30'
  notes: Optional notes about parity, caveats, etc.
```

The exact enum values for every field live in `schema/model.schema.json` and
are mirrored in `templates/model-entry.yaml`.

### 2. Add artifact provenance to `artifacts.yaml`

Append an entry under `artifacts:` **and bump `metadata.count` by 1** at the
top of the file — the audit fails when `metadata.count` does not equal the
actual number of artifact entries.

<!-- doc-test: artifact -->
```yaml
- id: my-model-1b
  group: zoo
  github:
    owner: converter-org
    repo: coreai-conversions
    path: https://github.com/converter-org/coreai-conversions/blob/main/zoo/my-model.md
  huggingface:
    owner: hf-user
    repo: MyModel-1B-CoreAI
    url: https://huggingface.co/hf-user/MyModel-1B-CoreAI
    # path: tree/main/vl       # optional sub-path within the HF repo
  officiality:
    apple_export_recipe: false
    apple_hosted_artifact: false
    community_packaged: true
```

Provenance notes:

- At least one of `github` / `huggingface` must be present. For an HF-only
  conversion (no GitHub repo), omit the `github` block entirely — **never
  fabricate GitHub coordinates** to satisfy a template.
- Optional content-addressing and provenance fields (`huggingface.revision`,
  `huggingface.files[]` with per-file `sha256`, `provenance.converted_by`,
  `provenance.recipe_source`, `provenance.format_version`, `mirrors[]`) let
  consumers pin and verify exact bytes. Fill them when you know the values;
  leave them absent rather than guessing — this catalog never fabricates data.
  See `templates/artifact-entry.yaml` for the full field list.

### 3. Add source/upstream records (if missing)

If the converter, original model, or artifact host is not already in
`sources.yaml` or `upstreams.yaml`, add it in the same PR.

### 4. Do NOT add benchmarks in a model PR

Benchmarks are a separate lane (see [Submit a benchmark](#submit-a-benchmark)
below). A model PR that also touches `benchmarks.jsonl` will fail the
benchmark workflow's one-line check and cannot merge.

## Contributing a new conversion

If the model you want listed has no `.aimodel` conversion yet, convert it
through **[coreai-fabric](https://github.com/kevinqz/coreai-fabric)** — the
first-party, agent-first conversion pipeline:

1. **Recipe** — pick or author a conversion recipe.
2. **Convert** — run the pipeline to produce the `.aimodel` bundle.
3. **Verify** — run the parity/verification gates.
4. **Publish** — upload to **your own** Hugging Face repo (the catalog indexes,
   it never hosts weights).
5. **Register** — open a model-lane PR here with the `catalog.yaml` +
   `artifacts.yaml` entries (fabric emits draft entries for you).

The community zoo (`john-rocky/coreai-model-zoo`) remains an **indexed
reference upstream**: the catalog tracks its conversions, but new
contributions should flow through coreai-fabric.

## Officiality rules

- **`apple_export_recipe: true`** — use **only** for conversions that follow Apple's official export recipes from `apple/coreai-models`.
- **`apple_hosted_artifact: true`** — use **only** if Apple hosts the artifact themselves (currently always `false`).
- **`community_packaged: true`** — use for community-packaged artifacts (e.g., `john-rocky/coreai-model-zoo` ports hosted on Hugging Face by `mlboydaisuke`, or conversions published via coreai-fabric).
- `source_group` must be consistent with `officiality`:
  - `source_group: official` → `apple_export_recipe: true`
  - `source_group: zoo` → `apple_export_recipe: false`
  - `source_group: fabric` → conversions produced and verified via the coreai-fabric pipeline

## Submit a benchmark

Benchmarks are **append-only JSONL**: one JSON object per line in
`benchmarks.jsonl`. Never overwrite or delete existing rows. Each benchmark PR
must add **exactly one line** and touch no other file.

### Example entry

<!-- doc-test: benchmark -->
```json
{"id": "my-model-1b-a18pro-gpu-toks", "model_id": "my-model-1b", "metric": "decode_throughput", "unit": "tokens_per_second", "value": 52.3, "device_class": "A18 Pro", "os_major": "27", "compute_unit": "GPU", "extraction_method": "upstream_readme_manual", "device_verified": false, "observed_date": "2026-06-30", "source": "coreai-model-zoo-readme", "confidence": "medium", "notes": "Decode throughput from upstream README table."}
```

Required fields (see `schema/benchmark.schema.json` for enums): `id`,
`model_id`, `metric`, `value`, `unit`, `device_class`, `os_major` (a
**string**, e.g. `"27"`), `compute_unit`, `extraction_method`, `confidence`,
`observed_date` (date only, `YYYY-MM-DD`), `source`.

### Two submission paths

**1. Signed relay path (not yet public).** Device-run measurements are meant
to flow through a privacy relay that coarsens device identifiers and adds an
Ed25519 `_signature` field to the JSONL line. CI verifies that signature and,
if every tier gate passes (gate outcome `signed_plausible` — the exact list
is in GOVERNANCE.md), merges automatically. **The relay is not
yet publicly deployed** — you cannot produce a valid `_signature` yourself, and
hand-crafted signatures are rejected. Do not attempt this path today.

**2. Curator review lane (open to everyone today).** Unsigned entries whose
`extraction_method` is `upstream_readme_manual` or `upstream_readme_scripted`
(i.e., numbers transcribed from an upstream README) are accepted for **human
curator review**: CI validates the line, applies the
`benchmark-curator-review` label, posts a summary comment, and passes. A
curator then reviews the provenance and merges manually. This is the normal
path for community benchmark contributions right now.

Unsigned entries with any other `extraction_method` are rejected — they claim
a measurement channel that requires the signed relay.

## Required checks

Before submitting a pull request, run **all** of these and ensure zero errors:

```bash
python scripts/validate.py
python scripts/audit.py
python scripts/doc_test.py
python scripts/generate.py
```

Then verify no generated files are out of sync:

```bash
git diff --exit-code docs/ dist/ templates/
```

CI runs the same checks, so if they pass locally they will pass in CI.
`scripts/doc_test.py` validates every tagged example in this file (and the
`templates/`) against the JSON Schemas, so documentation cannot drift from the
enforced contract. Remember the `artifacts.yaml` `metadata.count` bump — the
audit checks it.

## Unknowns

Use `unknown` instead of guessing. The audit enforces zero `unknown` values in critical fields (`precision`, `quantization`, `artifact_size`, runtime flags). If the upstream doesn't disclose a value, use `not_published` instead of `unknown`. (Exception: `artifact.availability` has no `not_published` enum value — its only fallback is `unknown`.)

## Submitting

1. Fork the repository.
2. Create a branch: `git checkout -b add-my-model`.
3. Make changes to source files only (YAML, or one `benchmarks.jsonl` line — never both in one PR).
4. Run all checks above.
5. Commit source changes + regenerated derived files.
6. Open a pull request describing what was added.
